## 二、选择模块：数据处理流程（Data Processing Pipeline）

### 2.1 现有实现逻辑梳理

**当前状态：** 仅有数据模型定义和示例，无实际实现。

**参考案例（来自 examples/default/data.md）：**

以 `garment-factory-cleaner` 项目为例，展示标准数据工作流：

```
garment-factory-cleaner/data/
├── raw/            # 原始数据（从 OSS 下载）
│   ├── 半年产量数据/
│   ├── 工序表/
│   ├── 返工数量/
│   └── 考勤/
├── cleaned/        # 处理后待验收数据（脚本输出）
└── final/          # 验收后数据（云端同步）
```

**数据生命周期：**
```
raw → cleaned → final
原始 → 处理后 → 已验收
```

**Pipeline 定义（来自 docs/drd/data.md）：**

工厂产量数据处理流程包含 5 个步骤：

| 步骤 | Task | Dataset（输入→输出） | 说明 |
|------|------|----------------------|------|
| 导入 | `import-raw-output` | 外部文件 → `raw-output-data` | 导入工厂原始产量数据 |
| 清洗 | `cleanse-output-data` | `raw-output-data` → `cleaned-output-data` | 修正负数、处理缺失值、过滤异常 |
| 合并 | `merge-output-records` | `cleaned-output-data` → `merged-output-records` | 整合多源数据，去重消歧 |
| 计算 | `compute-output-kpi` | `merged-output-records` → `output-analysis-results` | 计算日产量、合格率、效率 |
| 报表 | `generate-analysis-report` | `output-analysis-results` → 报告 | 生成趋势分析、利用率报告 |

**数据状态机：**

DatasetStatus 枚举：
- `pending`：等待中
- `ready`：已就绪
- `outdated`：已过时（上游数据更新后自动标记）
- `failed`：异常

### 2.2 优化方向建议

**建议补充：自动化数据质量评估与报告生成（Automated Data Quality Assessment & Reporting）**

**为什么需要这个功能？**

从业务场景文档可以看出，当前痛点是：
> "数据处理项目交付仅有最终结果，缺乏过程文档。客户不了解数据来源、处理方法、**质量评估**。"（docs/brd/scenario/delivery.md）

当前的 Pipeline 只记录了"**做了什么处理**"，但没有量化"**处理后的数据质量如何**"。

**数据质量评估的价值：**

1. **建立信任**：用客观指标证明数据可靠性，而非口头承诺
2. **快速定位问题**：当客户质疑数据准确性时，能快速回溯到具体质量维度
3. **标准化交付**：形成可复用的质量评估体系，降低人工编写报告的成本
4. **支撑溢价**：专业的质量报告能体现数据工程能力，支撑更高报价

**评估维度设计：**

基于数据分析最佳实践，建议从以下 6 个维度评估数据质量：

| 维度 | 指标 | 说明 | 示例 |
|------|------|------|------|
| **完整性** | 缺失率 | 关键字段空值比例 | 工厂名称缺失率 < 0.1% |
| **准确性** | 异常值比例 | 超出合理范围的记录占比 | 日产量 > 10000 件的记录占 0.5% |
| **一致性** | 跨表一致性 | 关联表之间的数据匹配度 | ERP 系统与现场记录匹配率 98% |
| **唯一性** | 重复率 | 完全重复或近似重复记录比例 | 重复订单记录占 0.2% |
| **及时性** | 数据新鲜度 | 最新数据距离当前的时间 | 最近一次更新在 2 小时前 |
| **规范性** | 格式合规率 | 符合预定义格式的记录比例 | 日期格式错误率 < 0.01% |

```python
# Provider 端模型扩展

from enum import Enum
from pydantic import BaseModel, Field
from typing import Optional

class QualityDimension(str, Enum):
    COMPLETENESS = "completeness"      # 完整性
    ACCURACY = "accuracy"              # 准确性
    CONSISTENCY = "consistency"        # 一致性
    UNIQUENESS = "uniqueness"          # 唯一性
    TIMELINESS = "timeliness"          # 及时性
    VALIDITY = "validity"              # 规范性


class AnomalyDetail(BaseModel):
    """异常详情"""
    dimension: QualityDimension
    severity: str  # high/medium/low
    description: str
    affected_rows: int
    sample_records: list[dict]  # 前 10 条异常样本
    suggestion: str  # 修复建议


class QualityReport(BaseModel):
    """数据质量报告"""
    id: str
    dataset_id: str
    overall_score: float  # 0-100
    dimension_scores: dict[QualityDimension, float]
    total_rows: int
    anomaly_count: int
    anomalies: list[AnomalyDetail]
    generated_at: datetime
    metadata: dict = {}  # 评估参数、耗时等


class QualityAssessmentRequest(BaseModel):
    """质量评估请求"""
    dataset_id: str
    rules: Optional[dict] = None  # 自定义规则覆盖
    sample_size: Optional[int] = None  # 采样大小（大数据集用）
```

#### 质量评估引擎设计

```python
class QualityAssessor:
    """数据质量评估器"""
    
    def __init__(self, dataset_path: Path):
        self.dataset_path = dataset_path
        self.df = self._load_dataset()
    
    def assess_completeness(self, required_columns: list[str]) -> float:
        """评估完整性"""
        missing_counts = self.df[required_columns].isnull().sum()
        total_cells = len(self.df) * len(required_columns)
        missing_cells = missing_counts.sum()
        return (1 - missing_cells / total_cells) * 100
    
    def assess_accuracy(self, rules: dict) -> tuple[float, list[AnomalyDetail]]:
        """评估准确性"""
        anomalies = []
        
        # 示例：检查负数产量
        if "negative_production" in rules:
            negative_rows = self.df[self.df['production_qty'] < 0]
            if len(negative_rows) > 0:
                anomalies.append(AnomalyDetail(
                    dimension=QualityDimension.ACCURACY,
                    severity="high",
                    description=f"发现 {len(negative_rows)} 条负数产量记录",
                    affected_rows=len(negative_rows),
                    sample_records=negative_rows.head(10).to_dict('records'),
                    suggestion="检查数据采集脚本，确认是否为录入错误"
                ))
        
        accuracy_score = max(0, 100 - len(anomalies) * 10)
        return accuracy_score, anomalies
    
    def assess_consistency(self, reference_dataset: Path) -> float:
        """评估一致性（与参考数据集对比）"""
        ref_df = pd.read_excel(reference_dataset)
        
        # 示例：检查款号匹配率
        matched = self.df.merge(ref_df, on='style_code', how='inner')
        match_rate = len(matched) / len(self.df) * 100
        
        return match_rate
    
    def generate_report(self) -> QualityReport:
        """生成完整质量报告"""
        scores = {}
        all_anomalies = []
        
        # 执行各维度评估
        scores[QualityDimension.COMPLETENESS] = self.assess_completeness(
            required_columns=['factory_name', 'date', 'style_code', 'quantity']
        )
        
        accuracy_score, accuracy_anomalies = self.assess_accuracy(
            rules={'negative_production': True, 'excessive_output': True}
        )
        scores[QualityDimension.ACCURACY] = accuracy_score
        all_anomalies.extend(accuracy_anomalies)
        
        # ... 其他维度评估
        
        # 计算综合得分（加权平均）
        weights = {
            QualityDimension.COMPLETENESS: 0.25,
            QualityDimension.ACCURACY: 0.30,
            QualityDimension.CONSISTENCY: 0.20,
            QualityDimension.UNIQUENESS: 0.10,
            QualityDimension.TIMELINESS: 0.10,
            QualityDimension.VALIDITY: 0.05
        }
        overall_score = sum(scores[dim] * weights[dim] for dim in scores)
        
        return QualityReport(
            id=f"qr_{uuid4().hex[:8]}",
            dataset_id=self.dataset_id,
            overall_score=round(overall_score, 2),
            dimension_scores={k.value: round(v, 2) for k, v in scores.items()},
            total_rows=len(self.df),
            anomaly_count=len(all_anomalies),
            anomalies=all_anomalies,
            generated_at=datetime.now(timezone.utc)
        )
```

#### 集成到 Pipeline

在数据处理流程中自动触发质量评估：

```python
# Pipeline 执行器伪代码

class PipelineExecutor:
    def execute_step(self, step: PipelineStep):
        # 1. 执行数据处理
        output_dataset = step.processor.run(step.input_datasets)
        
        # 2. 自动触发质量评估
        assessor = QualityAssessor(output_dataset.path)
        report = assessor.generate_report()
        
        # 3. 保存报告
        self.save_quality_report(report)
        
        # 4. 如果质量不达标，发出警告
        if report.overall_score < 80:
            self.notify_team(f"数据集 {output_dataset.name} 质量评分偏低: {report.overall_score}")
        
        # 5. 更新数据集状态
        output_dataset.quality_report_id = report.id
        output_dataset.status = "ready" if report.overall_score >= 85 else "needs_review"
```

## 三、业务目标推测

### 3.1 核心业务目标判断

基于对文档的分析，我认为量潮数据团队当前最核心的业务目标是：

**「建立数据交付服务的标准化产品能力，支撑规模化扩张」**

**具体拆解为三个关键问题：**

1. **信任成本过高**
   - 现状：客户靠群聊了解进度，信息不透明
   - 目标：通过产品化界面降低沟通成本，建立专业形象
   - 证据：PMD 明确指出"核心矛盾是信息不透明，不是操作效率低"

2. **交付过程黑盒化**
   - 现状：只有最终结果，客户不知道中间发生了什么
   - 目标：让数据处理过程可追溯、可验证
   - 证据：BRD 场景五强调"数据工程规范"和"交付过程透明"

3. **无法规模化获客**
   - 现状：依赖校友圈和知乎大V，渠道有限
   - 目标：通过产品化能力形成口碑传播，突破地域限制
   - 证据：BRD 场景二提到"产能不足，无法扩大市场规模"

**业务发展阶段判断：**
- 处于**从手工服务向产品化转型**的关键期
- 已有稳定客户群体（985高校科研工作者），但交付方式仍依赖人工
- 需要通过技术手段将服务能力标准化、可复制化

### 3.2 核心挑战

1. **如何在不增加运营负担的前提下提升透明度？**
   - 不能要求数据工程师手动录入大量信息
   - 需要自动化采集项目进度和资产状态

2. **如何平衡"只读展示"与"客户期望"？**
   - PMD 明确当前只做信息展示
   - 但客户可能会要求在系统内完成验收、反馈等操作
   - 需要预留扩展空间

3. **如何证明专业性以支撑溢价？**
   - 目标客户有充足经费（每年几万到十几万）
- 需要通过产品细节体现数据工程的专业度
   - **质量评估报告是关键差异化点**

## 四、方案规划思路

### 4.1 业务目标 → 技术方向拆解

**目标：建立标准化产品能力，支撑规模化**

**技术拆解矩阵（数据分析视角）：**

| 业务诉求 | 技术方向 | 优先级 | AI 可承担部分 | 人工把控重点 |
|---------|---------|--------|--------------|-------------|
| 信息透明化 | 实时进度同步 API + 前端看板 | P0 | API 脚手架、UI 组件模板 | 业务状态机设计、异常场景处理 |
| **过程可验证** | **自动化数据质量评估** | **P0** | **评估规则代码生成、统计计算** | **指标体系设计、阈值设定** |
| 降低运营成本 | 自动化报告生成（PDF/Excel） | P1 | 报告模板渲染、图表生成 | 报告叙事逻辑、关键洞察提炼 |
| 建立专业形象 | 数据血缘追踪 + 影响分析 | P1 | 图谱构建算法、可视化组件 | 血缘语义定义、业务规则映射 |
| 支撑规模化 | 可复用评估规则库 | P2 | 规则 DSL 解析器、测试用例 | 规则有效性验证、领域知识沉淀 |

### 4.2 AI 协作分工原则

**适合交给 AI 执行的工作：**

1. **标准化代码生成**
   - 质量评估规则的 Python 实现（如缺失率计算、异常值检测）
   - 统计指标计算函数（均值、方差、分位数等）
   - 数据可视化代码（matplotlib/seaborn 图表）
   - 单元测试骨架

2. **重复性配置工作**
   - Pandas DataFrame 处理 pipeline
   - 数据库 schema 迁移脚本
   - API 路由和序列化模型

3. **文档辅助**
   - 根据代码自动生成 API 文档
   - 质量报告模板设计
   - 技术方案对比分析

4. **数据分析辅助**
   - 探索性数据分析（EDA）代码生成
   - 异常模式识别
   - 性能瓶颈诊断

**需要人工重点把控的环节：**

1. **业务语义定义**
   - 什么是"合理的产量范围"？（需要领域知识）
   - 哪些字段是"关键字段"必须非空？
   - 质量评分的权重如何分配？

2. **阈值设定**
   - 完整性 95% 算合格还是 90%？
   - 什么情况下触发告警？
   - 不同行业是否需要不同的标准？

3. **异常处理策略**
   - 发现负数产量时，是自动修正还是标记待审核？
   - 如何处理缺失值：删除、填充、还是插值？
   - 业务规则冲突时的优先级

4. **用户体验校验**
   - 质量报告的易读性
   - 异常描述是否清晰易懂
   - 修复建议是否具有可操作性

5. **安全性审查**
   - 敏感数据脱敏（如客户名称、金额）
   - 访问权限控制
   - 审计日志记录

### 4.3 确保方案对齐业务目标的机制

**1. 建立"问题-方案"映射表**

每次技术决策前，强制回答三个问题：
- 这个功能解决了哪个业务问题？（引用 BRD/PMD 中的具体场景）
- 如果不做这个功能，业务会受到什么影响？
- 有没有更简单的替代方案？

**示例：**
> 问：为什么要做自动化数据质量评估？  
> 答：解决 BRD 场景五中"客户不了解质量评估"的问题。如果不做，客户信任度下降，项目验收延期。替代方案是人工编写质量报告，但会增加运营成本且容易出错。

**2. 采用"最小可行质量评估"策略**

- 第一版只实现 3 个核心维度（完整性、准确性、一致性）
- 根据客户反馈逐步增加维度（唯一性、及时性、规范性）
- 避免一次性过度设计

**3. 设置业务指标监控**

技术指标必须关联业务指标：
- 质量评估覆盖率 > 90% → 客户感知到的专业性
- 报告生成时间 < 30秒 → 不影响交付效率
- 异常检出准确率 > 85% → 减少客户投诉

**4. 定期业务回顾会议**

每两周与业务方（交付团队/客户代表）对齐：
- 客户对质量报告的反馈如何？
- 哪些异常被误判？需要调整规则
- 是否有新的质量维度需要加入？

**5. 文档驱动开发（DDD 变体）**

- 所有新功能先在 BRD 中描述业务问题
- 技术方案在 DRD（数据需求文档）中评审
- 代码实现后反向更新文档，保持一致性

---

## 五、总结

**我的核心观点：**

量潮数据正处于**从服务型公司向产品型公司转型**的关键节点。当前的技术投入不应追求大而全的功能堆砌，而应聚焦于**用最低成本建立客户信任**。

**自动化数据质量评估**是我选择的切入点，因为：
1. 它直接回应了"交付过程黑盒化"的核心痛点
2. 它能体现数据工程的专业度，支撑溢价能力
3. 它的技术复杂度适中，可在 2-3 周内交付 MVP
4. 它为后续的自动化报告、智能推荐等功能打下数据基础

**AI 协作的关键：**

不要试图用 AI 替代思考，而是用 AI 放大执行力。业务语义、阈值设定、异常处理策略必须由人主导；代码生成、统计计算、文档辅助可以充分授权给 AI。

**最后的原则：**

> 先让信息流动起来，再考虑如何优化流动的效率。

这与 PMD 中"先解决透明度，再考虑操作效率"的判断完全一致。

**作为数据分析师的价值：**

在这个项目中，数据分析师不仅是"跑脚本的人"，更是：
1. **质量标准的制定者**：定义什么是"好数据"
2. **业务规则的翻译者**：将领域知识转化为可执行的评估规则
3. **客户信任的建立者**：通过专业的质量报告赢得客户认可

这正是量潮数据从"外包服务商"升级为"数据工程专家"的关键所在。
